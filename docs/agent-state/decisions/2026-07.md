## D196 - D195 이후 다음 단위는 production pump 변경이 아니라 fixed-write socket fd contract evidence 다.

- Date: 2026-07-07
- Status: Accepted
- Scope: Phase 6 `io_uring` post-D195 next scope.

### Context

D195에서 D181 fixed-write SQE helper/native completion evidence 가 원격
`iouring-linux-contract.yml` run `28834265348`에서 통과했다. 해당 evidence 는 registered buffer slice 를
anonymous pipe fd 로 `IORING_OP_WRITE_FIXED` 제출하고 completion result 2와 payload `{20,30}`을 확인한 것이다.

이 결과는 fixed-write opcode/field mapping 과 kernel completion 계약을 닫지만, production TCP/UDP pump 가
fixed buffer 를 써도 된다는 직접 근거는 아니다. 특히 socket fd 지원 여부, `RefCountedBuffer` registration lifetime,
TCP length prefix, in-flight close drain, fallback 정책이 아직 분리되어 있지 않다.

### Decision

D195 이후 다음 구현 단위는 production pump 변경이 아니라 **fixed-write socket fd contract evidence** 로 둔다.

- `tests/Hps.Transport.IoUring.Tests`에 Linux capability gated native test 를 추가한다.
- test-only `socketpair(AF_UNIX, SOCK_STREAM)` helper 를 사용한다.
- 한쪽 stream socket fd 에 registered buffer `{10,20,30,40}` offset 1 length 2를 `TrySubmitWriteFixed`로 제출한다.
- 반대쪽 socket fd 에서 `{20,30}`을 읽고 completion result 2를 검증한다.
- production `IoUringTransport`, `TransportConnection`, `RefCountedBuffer`, `ITransport` public surface 는 변경하지 않는다.

### Rationale

pipe evidence 다음에 바로 TCP pump 를 바꾸면 kernel/socket contract 와 transport ownership 문제가 동시에 섞인다.
`socketpair` evidence 는 실제 pump 를 건드리지 않고 stream socket fd 에 `WRITE_FIXED`가 적용되는지를 작고 결정적으로
검증한다. 이는 향후 TCP pump fixed-buffer 설계에서 실패 원인을 socket contract 와 ownership/lifetime 문제로 분리하는 기준점이 된다.

### Consequences

- D196 구현 후에도 fixed-buffer TCP pump 연결은 별도 설계와 TDD 계획이 필요하다.
- zero-copy send, notification CQE, fixed-buffer lease/cache owner, default promotion, latency hard gate 는 계속 보류한다.
- D196 remote gate 는 사용자 push 이후 `iouring-linux-contract.yml` artifact 로 capability `Available` 상태의 socket fixed-write test 통과를 확인해야 닫힌다.

## D194 - `iouring-linux-contract.yml`은 Linux에서 solution restore/build가 아니라 `Hps.Transport.IoUring.Tests` project restore/build/test만 수행한다.

- Date: 2026-07-07
- Status: Accepted
- Scope: Linux io_uring contract workflow restore/build boundary.

### Context

D183~D192로 WPF sample dashboard가 solution에 포함됐다. WPF project는 `net9.0-windows`/`UseWPF`를 사용하므로,
Linux runner에서 solution-level `dotnet restore HighPerformanceSocket.slnx`를 실행하면
`EnableWindowsTargeting`이 필요하다는 `NETSDK1100` 오류가 발생한다.

D181 원격 fixed-write evidence gate의 목적은 WPF sample 또는 solution 전체 검증이 아니라
`Hps.Transport.IoUring.Tests`가 Linux capability available 환경에서 native syscall contract를 통과하는지 확인하는 것이다.

### Decision

`iouring-linux-contract.yml`의 restore/build 범위를 `tests/Hps.Transport.IoUring.Tests/Hps.Transport.IoUring.Tests.csproj`로 제한한다.

- Restore: `dotnet restore tests/Hps.Transport.IoUring.Tests/Hps.Transport.IoUring.Tests.csproj`
- Build: `dotnet build tests/Hps.Transport.IoUring.Tests/Hps.Transport.IoUring.Tests.csproj --no-restore`
- Test: 기존처럼 같은 test project를 `--no-build --no-restore`로 실행한다.
- Linux workflow에 `EnableWindowsTargeting`을 추가하지 않는다.

### Rationale

contract workflow는 io_uring backend의 Linux native evidence 수집 경로다. solution 전체를 Linux에서 restore/build하면
Windows-only WPF sample의 플랫폼 제약이 io_uring contract gate를 가로막는다.
`EnableWindowsTargeting`을 켜는 방식은 증상을 우회할 수 있지만, workflow가 검증하지 않아야 할 WPF project까지
Linux contract gate 범위에 포함시키므로 경계가 흐려진다.

### Consequences

- WPF sample dashboard가 solution에 포함되어도 `iouring-linux-contract.yml`은 Linux에서 독립적으로 실행될 수 있다.
- solution-level build/test 검증은 로컬 Windows 또는 별도 CI 경로의 책임으로 유지한다.
- D181 fixed-write evidence gate는 workflow fix push 이후 새 원격 run artifact에서 다시 확인해야 한다.

## D183 — 실행 검증용 sample dashboard 는 WinUI 3가 아니라 WPF/MVVM 으로 만들고 production transport 정책과 분리한다.

- Date: 2026-07-06

후보는 WPF 와 WinUI 3였지만, 이 저장소의 목적은 UI framework adoption 이 아니라 고성능 socket Interface Server 의 실행 검증이다.

### Decision

샘플 대시보드는 WPF/MVVM 으로 만든다.

- 프로젝트는 `samples/Hps.Sample.Dashboard`에 둔다.

### Rationale

WPF 는 .NET SDK 기반 실행 경로가 단순하고, WinUI 3/Windows App SDK packaging/runtime 전제를 새로 추가하지 않는다.

### Consequences

- 다음 단계는 `docs/superpowers/plans/2026-07-06-wpf-sample-dashboard.md` 구현 계획 작성이다.
- public API 부족분이 발견되면 UI 편의용으로 즉시 넓히지 않고 별도 task 로 분리한다.
- WinUI 3, WPF 앱 내부 Linux native `io_uring` 실행, fixed-buffer pump 연결, zero-copy send 는 이번 샘플 범위 밖이다.

## D181 — fixed-write SQE helper 는 evidence-only boundary 로 추가하고 TCP/UDP pump 연결은 원격 contract gate 이후로 둔다.

- Date: 2026-07-03
- Status: Accepted
- Scope: Phase 6 `io_uring` fixed-buffer SQE submission local evidence.

### Context

D179/D180은 D178 register/unregister evidence 이후 다음 최소 단위를 fixed-buffer SQE submission contract evidence 로 정했다.
이번 구현은 `OperationWriteFixed`와 `TrySubmitWriteFixed(...)`를 추가하고,
Linux capability available 환경에서 registered buffer slice 를 pipe write fd 로 제출하는 test 를 만든다.

### Decision

`TrySubmitWriteFixed(...)`는 evidence-only helper 로 추가한다.

- helper 는 SQE `Opcode`, `FileDescriptor`, `Address`, `Length`, `BufferIndex`, `UserData`를 채운다.
- helper 는 production TCP/UDP pump 에 연결하지 않는다.
- local/Windows 에서는 native evidence test 가 capability guard 로 빠진다.
- 원격 `iouring-linux-contract.yml`에서 capability available 상태 completion evidence 를 확인한 뒤 다음 단위를 판단한다.

### Rationale

fixed-buffer operation 의 첫 위험은 registration 자체가 아니라 SQE field mapping 과 kernel completion 계약이다.
production pump 에 바로 연결하면 ref-count, length-prefix, fan-out, close drain 문제가 동시에 섞인다.
따라서 helper 를 먼저 evidence boundary 로 만들고, 원격 Linux contract gate 를 통과한 뒤 pump integration 여부를 판단한다.

### Consequences

- D181 local evidence 는 TCP/UDP pump behavior 를 바꾸지 않는다.
- 다음 작업은 원격 `iouring-linux-contract.yml` artifact gate 검토다.
- fixed-buffer pump, zero-copy send, notification CQE, default promotion, latency hard gate 는 계속 보류한다.

## D179 — D178 이후 다음 단위는 pump 변경이 아니라 fixed-buffer SQE submission contract evidence 다.

- Date: 2026-07-03
- Status: Accepted
- Scope: Phase 6 `io_uring` fixed-buffer SQE submission contract evidence.

### Context

D178은 `IoUringRegisteredBufferSet`이 Linux runner 에서 fixed buffer register/unregister 를 수행할 수 있음을 확인했다.
하지만 현재 TCP send path 는 `IORING_OP_SEND`, UDP send path 는 `IORING_OP_SENDMSG`를 사용하고 있으며,
registered buffer index 를 SQE에 넣어 실제 I/O에 쓰는 경로는 아직 없다.

man7/liburing 문서 기준 fixed-buffer I/O는 등록된 buffer index 와 등록 영역 안의 address/length 를 함께 요구한다.
또한 `send_zc_fixed`는 일반 send 와 달리 memory reuse notification CQE 를 추가로 처리해야 한다.

### Decision

D178 이후 다음 구현 단위는 production pump 변경이 아니라 fixed-buffer SQE submission contract evidence 로 둔다.

- `IoUringNative`에 fixed-write opcode 를 추가한다.
- `IoUringQueue`에 fixed-buffer write submission helper 를 추가한다.
- 테스트는 shape test 와 Linux/capability gated native completion test 로 나눈다.
- native completion test 는 pump 와 분리된 작은 local fd 를 사용한다.
- `iouring-linux-contract.yml`은 이미 `Hps.Transport.IoUring.Tests` 전체를 실행하므로 workflow 는 바꾸지 않는다.

### Rationale

fixed-buffer pump 연결은 `TransportConnection.InFlightSend`, length-prefix block, `RefCountedBuffer` fan-out,
close drain, registration lifetime 을 한 번에 엮는다. zero-copy send 는 여기에 notification CQE 기반 memory reuse 계약까지 더한다.
따라서 다음 최소 증거는 SQE opcode/field/register range/completion 경계를 분리해서 검증하는 것이다.

### Consequences

- 다음 구현은 `IORING_OP_WRITE_FIXED` 제출 shape/helper 와 Linux native completion evidence 에 집중한다.
- TCP/UDP pump fixed-buffer 연결은 이 contract evidence 이후 별도 설계로 판단한다.
- zero-copy send, notification CQE 처리, default backend promotion, latency hard gate 는 계속 보류한다.

## D177 — fixed buffer registration owner 는 RegisteredBufferCount evidence 와 Linux native register/unregister test 를 가진다.

- Date: 2026-07-03
- Status: Accepted
- Scope: Phase 6 `io_uring` fixed buffer registration evidence implementation.

### Context

D176은 다음 단위를 fixed/zero-copy pump 연결이 아니라 `IoUringRegisteredBufferSet` Linux native register/unregister evidence 로 정했다.
원격 artifact 에서 register/unregister 가 실제로 몇 개 buffer 를 대상으로 수행됐는지 검증 가능한 관측값이 필요했다.

### Decision

`IoUringRegisteredBufferSet`에 internal `RegisteredBufferCount`를 추가하고,
`IoUringRegisteredBufferSetTests`에 Linux/capability gated native register/unregister test 를 둔다.

- Windows/non-Linux 는 기존 platform guard 를 유지한다.
- Linux capability available 환경에서는 `IoUringQueue.CreateForProbe(4)`로 작은 ring 을 만들고
  2개 byte[] buffer 를 `IoUringRegisteredBufferSet.Register(...)`로 등록한 뒤 dispose 한다.
- test output 에 capability status 와 registered fixed buffer count 를 남긴다.

### Rationale

fixed-buffer pump 를 바로 연결하기 전, native registration owner 가 Linux runner 에서 실제 register/unregister 를 수행할 수 있음을
분리된 artifact 로 확인해야 한다.
`RegisteredBufferCount`는 hot path API가 아니라 internal evidence surface 이며, 후속 fixed-buffer pump 가 기대하는
buffer table 크기 계약을 테스트에서 명시할 수 있게 한다.

### Consequences

- production TCP/UDP pump 동작은 바뀌지 않는다.
- 다음 작업은 push 이후 `iouring-linux-contract.yml`을 실행해 TRX/summary 에서 native evidence test 실행을 확인하는 것이다.
- fixed-buffer SQE 연결, zero-copy send, default promotion, latency hard gate 는 계속 별도 결정 전까지 보류한다.

## D174 — io_uring completion loop 는 shutdown 이후 unregister 된 token 의 늦은 CQE만 stale completion 으로 흡수한다.

- Date: 2026-07-02
- Status: Accepted
- Scope: Phase 6 `io_uring` TCP benchmark stop reliability fix.

### Context

D173 reference 확장 이후 원격 `iouring-benchmark-artifacts.yml` run `28570636117`은 UDP baseline/summary/history/envelope 는
모두 exit 0이었지만 TCP baseline suite 가 exit 134로 실패했다.
로그의 직접 원인은 `IoUringOperationRegistry.Resolve()`의
`등록되지 않은 io_uring operation token입니다.` 예외였다.
스택은 `IoUringTransport.StopCore()`가 connection 을 닫는 중 resource context 를 unregister 한 뒤,
completion loop 가 늦게 도착한 CQE 를 drain 하면서 registry 에 없는 token 을 fatal mapping 오류로 처리한 흐름을 보여줬다.

### Decision

`IoUringCompletionLoop`에 shutdown 시작 경계를 명시한다.

- `IoUringTransport.StopCore()`는 connection/endpoint close 전에 `IoUringCompletionLoop.BeginShutdown()`을 호출한다.
- completion loop 는 shutdown 이후 registry 에 없는 token 의 CQE만 stale completion 으로 보고 no-op 처리한다.
- shutdown 전 registry 에 없는 token 은 기존처럼 `InvalidOperationException`으로 실패시켜 실제 token routing bug 를 숨기지 않는다.

### Rationale

Linux kernel 은 socket close/cancel/stop 경합에서 이미 완료됐거나 취소된 CQE 를 resource unregister 이후에 노출할 수 있다.
이 CQE는 새 managed context 로 라우팅하면 안 되지만, transport shutdown 을 실패시킬 오류도 아니다.
반대로 정상 운영 중 unknown token 을 항상 무시하면 user_data/token routing 손상을 감출 수 있으므로,
무시 정책은 shutdown 이후로만 제한해야 한다.

### Consequences

- TCP baseline open-loop stop 중 stale CQE가 남아도 process abort 로 이어지지 않아야 한다.
- 정상 운영 중 token corruption 은 계속 즉시 실패한다.
- 다음 작업은 fix push 이후 원격 `iouring-benchmark-artifacts.yml` artifact gate 를 재실행해
  TCP baseline exit 134 재발 여부와 TCP/UDP envelope signal-count 0을 확인하는 것이다.

## D147 — Linux io_uring benchmark artifact 는 별도 workflow_dispatch workflow 로 수집한다.

- Date: 2026-07-01
- Status: Accepted
- Scope: Phase 6 `io_uring` benchmark evidence path.

### Context

D146으로 benchmark CLI 는 `--backend iouring`을 지원하게 됐다.
하지만 기존 `.github/workflows/benchmark-artifacts.yml`은 Windows SAEA benchmark artifact 용이고,
`.github/workflows/iouring-linux-contract.yml`은 Linux `Hps.Transport.IoUring.Tests` 계약 검증용이다.
따라서 Linux available runner 에서 TCP/UDP `io_uring` benchmark raw report 를 생성하는 원격 evidence path 가 없다.

### Decision

`.github/workflows/iouring-benchmark-artifacts.yml`을 별도 workflow 로 추가한다.

- trigger 는 `workflow_dispatch` 전용으로 둔다.
- runner 는 `ubuntu-latest`로 둔다.
- runner identity 는 `ci-linux-iouring-x64-01`, kind 는 `ci`로 둔다.
- TCP와 UDP를 같은 workflow 에서 모두 실행하되 output root 는 protocol 별로 분리한다.
- protocol 별 root 바로 아래에 날짜 directory 를 둔다.
  즉 `runner/tcp/<yyyy-mm-dd>/session-01`, `runner/udp/<yyyy-mm-dd>/session-01` 구조를 사용한다.
  `BaselineHistoryReader`가 parent root 바로 아래 날짜 directory 만 읽기 때문이다.
- 각 protocol 은 `--baseline-suite <session-dir> --runs 1 --protocol <tcp|udp> --backend iouring`을 실행한다.
- 각 protocol 별 `summary.json`/`summary.md`와 `history.json`/`history.md`를 생성한다.
- root `summary.md`와 `dotnet-info.txt`를 함께 업로드한다.

### Rationale

기존 Windows benchmark workflow 를 matrix 로 넓히면 SAEA benchmark, CI baseline artifact, Linux `io_uring` evidence 가
같은 trigger와 artifact 정책에 섞인다. 현재 필요한 것은 default promotion 이 아니라 D146 selector 가 Linux runner 에서
실제 benchmark raw report 를 만들 수 있는지 확인하는 것이다. 별도 수동 workflow 가 가장 작고 추적 가능한 경계다.

### Consequences

- `io_uring` benchmark evidence 는 push 이후 사용자가 수동 workflow 를 실행해야 생긴다.
- artifact 는 `iouring-benchmark-artifacts-<yyyy-mm-dd>-github-<run-id>-<attempt>` 이름으로 올라간다.
- artifact 내부 history root 는 `tcp/`와 `udp/`이며, 각 root 아래 날짜 directory 와 session directory 가 위치한다.
- latency/HWM/warning 값은 report data 이며 hard gate 나 warning-as-failure 로 승격하지 않는다.
- default backend selection 은 계속 변경하지 않는다.
- fixed registration, zero-copy send, IPv6 direct UDP 는 이 artifact 를 검토한 뒤 별도 설계로 판단한다.
