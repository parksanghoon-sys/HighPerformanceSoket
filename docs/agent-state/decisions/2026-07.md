## D150 — io_uring `--runs 3` benchmark artifact 는 D149 evidence gate 를 충족했고 p99 warning 은 후속 분석으로 분리한다.

- Date: 2026-07-01
- Status: Accepted
- Scope: Phase 6 `io_uring` benchmark evidence review.

### Context

D149는 `iouring-benchmark-artifacts.yml`을 protocol 별 `--runs 3`으로 승격했다.
사용자 push 이후 run `28489104828`을 `workflow_dispatch`로 실행했고, workflow 는 success 로 완료됐다.

### Decision

run `28489104828` artifact 를 D149 evidence gate 충족으로 수락한다.

- artifact 는 `iouring-benchmark-artifacts-2026-07-01-github-28489104828-1`이다.
- root `summary.md`는 `Runs per protocol: 3`과 TCP/UDP baseline, summary, history exit code 0을 기록한다.
- TCP/UDP 각각 raw report 6개와 protocol 별 `summary.json`/`summary.md`, `history.json`/`history.md`가 존재한다.
- TCP summary 는 source-report-count 6, hard-passed true, warning-count 6,
  load p99 max 4570.8 us, open-loop p99 max 4604.5 us, dropped/payload-error/pool-rented 0,
  TCP HWM max 1이다.
- UDP summary 는 source-report-count 6, hard-passed true, warning-count 2,
  load p99 max 1506.4 us, open-loop p99 max 1349.3 us, dropped/payload-error/pool-rented 0,
  UDP HWM max 0이다.

### Rationale

D149의 gate 는 반복 artifact 생성과 delivery/drop/leak hard gate 확인이다. 해당 기준은 충족했다.
반면 warning 은 p99 latency soft signal 이며, 현재 threshold 는 runner/profile scoped io_uring reference envelope 가 아니라
기존 전역 기준에서 나온다. 따라서 warning-count 를 workflow failure 나 default promotion 차단으로 즉시 승격하지 않는다.

### Consequences

- D149 반복 benchmark artifact path 는 닫힌 것으로 본다.
- 다음 단위는 p99 warning 분석과 후속 방향 설계다.
- fixed payload registration cache, zero-copy send, latency threshold 조정, 반복 artifact 축적 중 어느 쪽이 우선인지
  D150 evidence 를 기준으로 다시 판단한다.
- latency hard gate, warning-as-failure, default backend promotion 은 계속 보류한다.

## D149 — io_uring benchmark artifact workflow 는 D148 이후 TCP/UDP baseline suite 를 `--runs 3`으로 승격한다.

- Date: 2026-07-01
- Status: Accepted
- Scope: Phase 6 `io_uring` benchmark evidence quality.

### Context

D148 run `28486254926`은 Linux `io_uring` TCP/UDP artifact path 를 닫았지만 protocol 별 `--runs 1` 표본이다.
이 표본은 raw report, summary, history 생성 경로 검증에는 충분하지만 TCP p99 warning 2개를 최적화 구현 근거로
바로 해석하기에는 부족하다.

### Decision

`iouring-benchmark-artifacts.yml`의 TCP/UDP baseline suite 를 `--runs 3`으로 승격한다.

- TCP command 는 `--baseline-suite "$BENCH_TCP_SESSION_DIR" --runs 3 --protocol tcp --backend iouring`을 사용한다.
- UDP command 는 `--baseline-suite "$BENCH_UDP_SESSION_DIR" --runs 3 --protocol udp --backend iouring`을 사용한다.
- root summary 는 `Runs per protocol: 3`을 표시한다.
- artifact layout, runner identity, failure policy, manual `workflow_dispatch` trigger 는 D147/D148과 동일하게 유지한다.

### Rationale

Windows benchmark workflow 는 이미 `--runs 3`을 사용한다. 같은 반복 수로 맞추면 io_uring artifact 도
session summary 안에서 load/open-loop 각 3개 raw report 를 집계하므로 p99/HWM 관측값을 단일 표본보다 안정적으로 볼 수 있다.
반면 fixed payload registration cache, zero-copy send, IPv6 direct UDP, default backend promotion 은
한 번짜리 artifact 성공만으로 열기에는 수명/호환성/운영 정책 범위가 크다.

### Consequences

- 다음 remote artifact 는 protocol 별 source-report-count 6을 기대한다.
- workflow 실행 시간은 늘지만 D148 run 1회가 2분대였고 timeout 40분 안에 충분히 들어간다.
- latency warning 은 계속 report-only 이며 hard gate 로 승격하지 않는다.
- 최적화 구현 여부는 `--runs 3` artifact 를 검토한 뒤 다시 판단한다.

## D148 — io_uring benchmark artifact workflow 는 protocol-first history root 구조로 D147 evidence gate 를 충족했다.

- Date: 2026-07-01
- Status: Accepted
- Scope: Phase 6 `io_uring` benchmark evidence review.

### Context

D147 workflow 의 첫 원격 run `28485295725`는 TCP/UDP baseline suite 와 summary 는 성공했지만,
history command 가 `baseline history summary.json 을 찾지 못했습니다`로 실패했다.
원인은 workflow output 이 `date/protocol/session` 구조라서 `BaselineHistoryReader`가 요구하는
"입력 root 바로 아래 날짜 directory" 규칙과 맞지 않은 것이다.

### Decision

D147 workflow artifact 구조 보정 후 run `28486254926` 결과를 D147 evidence gate 충족으로 수락한다.

- artifact 는 `iouring-benchmark-artifacts-2026-07-01-github-28486254926-1`이다.
- artifact 내부 protocol root 는 `tcp/`와 `udp/`이며, 각 root 바로 아래 `2026-07-01/session-01`이 위치한다.
- TCP/UDP baseline, summary, history exit code 는 모두 0이다.
- TCP summary 는 source-report-count 2, hard-passed true, warning-count 2,
  dropped-total 0, payload-error-total 0, pool-rented-max 0이다.
- UDP summary 는 source-report-count 2, hard-passed true, warning-count 0,
  dropped-total 0, payload-error-total 0, pool-rented-max 0이다.

### Rationale

이번 gate 의 목적은 Linux available runner 에서 `--backend iouring` TCP/UDP benchmark raw report,
summary, history 를 기존 schema 로 생성할 수 있는지 확인하는 것이다.
run `28486254926`은 TCP/UDP load/open-loop report 를 생성했고, protocol 별 summary/history reader 도
정상 집계했다. TCP p99 warning 은 D147 policy 상 evidence-only report data 이며,
delivery/drop/leak hard gate failure 가 아니다.

### Consequences

- D147의 원격 benchmark artifact path 는 닫힌 것으로 본다.
- TCP p99 warning 은 latency hard gate 나 warning-as-failure 로 승격하지 않는다.
- 다음 단위는 fixed payload registration cache, zero-copy send, IPv6 direct io_uring UDP,
  default backend promotion 중 지금 열어도 되는 최소 설계 후보를 재평가하는 것이다.
- 최적화나 default promotion 은 artifact 1회 성공만으로 바로 진행하지 않고 별도 결정으로 다룬다.

## D147 — Linux io_uring benchmark artifact 는 별도 workflow_dispatch workflow 로 수집한다.

- Date: 2026-07-01
- Status: Accepted
- Scope: Phase 6 `io_uring` benchmark evidence path.

### Context

D146으로 benchmark CLI 는 `--backend iouring`을 지원하게 됐다.
하지만 기존 `.github/workflows/benchmark-artifacts.yml`은 Windows SAEA benchmark artifact 용이고,
`.github/workflows/iouring-linux-contract.yml`은 Linux `Hps.Transport.IoUring.Tests` 계약 검증용이다.
따라서 Linux available runner 에서 TCP/UDP `io_uring` benchmark raw report 를 생성하는 원격 evidence path 가 없다.

### Decision

`.github/workflows/iouring-benchmark-artifacts.yml`을 별도 workflow 로 추가한다.

- trigger 는 `workflow_dispatch` 전용으로 둔다.
- runner 는 `ubuntu-latest`로 둔다.
- runner identity 는 `ci-linux-iouring-x64-01`, kind 는 `ci`로 둔다.
- TCP와 UDP를 같은 workflow 에서 모두 실행하되 output root 는 protocol 별로 분리한다.
- protocol 별 root 바로 아래에 날짜 directory 를 둔다.
  즉 `runner/tcp/<yyyy-mm-dd>/session-01`, `runner/udp/<yyyy-mm-dd>/session-01` 구조를 사용한다.
  `BaselineHistoryReader`가 parent root 바로 아래 날짜 directory 만 읽기 때문이다.
- 각 protocol 은 `--baseline-suite <session-dir> --runs 1 --protocol <tcp|udp> --backend iouring`을 실행한다.
- 각 protocol 별 `summary.json`/`summary.md`와 `history.json`/`history.md`를 생성한다.
- root `summary.md`와 `dotnet-info.txt`를 함께 업로드한다.
- baseline/summary/history 단계의 non-zero exit code 는 저장해 두고, upload 이후 final gate 에서 job failure 로 복원한다.

### Rationale

기존 Windows benchmark workflow 를 matrix 로 넓히면 SAEA benchmark, CI baseline artifact, Linux `io_uring` evidence 가
같은 trigger와 artifact 정책에 섞인다. 현재 필요한 것은 default promotion 이 아니라 D146 selector 가 Linux runner 에서
실제 benchmark raw report 를 만들 수 있는지 확인하는 것이다. 별도 수동 workflow 가 가장 작고 추적 가능한 경계다.

### Consequences

- `io_uring` benchmark evidence 는 push 이후 사용자가 수동 workflow 를 실행해야 생긴다.
- artifact 는 `iouring-benchmark-artifacts-<yyyy-mm-dd>-github-<run-id>-<attempt>` 이름으로 올라간다.
- artifact 내부 history root 는 `tcp/`와 `udp/`이며, 각 root 아래 날짜 directory 와 session directory 가 위치한다.
- latency/HWM/warning 값은 report data 이며 hard gate 나 warning-as-failure 로 승격하지 않는다.
- default backend selection 은 계속 변경하지 않는다.
- fixed registration, zero-copy send, IPv6 direct UDP 는 이 artifact 를 검토한 뒤 별도 설계로 판단한다.
