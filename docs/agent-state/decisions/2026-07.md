## D230 - registered payload pool local gate 는 통과했지만 Linux native fixed payload hit 는 remote artifact 로 확인한다.

- Date: 2026-07-10
- Status: Accepted
- Scope: Phase 6 `io_uring` registered payload pool local implementation gate.

### Context

D229 구현 계획에 따라 TCP publish payload 를 io_uring queue lifetime 에 묶인 registered block source 에서 대여하고,
send path 에서 backing array fixed index 를 찾아 `WRITE_FIXED`를 먼저 시도하는 경로를 구현했다.
Windows/local 환경에서는 io_uring capability 가 unavailable 이므로 native fixed payload hit 를 직접 실행할 수 없다.

### Decision

registered payload pool local gate 는 수락한다.

- `RefCountedBuffer`는 `IRefCountedBufferOwner`로 마지막 반환 owner 를 일반화한다.
- `TcpFrameAssembler`와 `TcpFrameReceiveHandler`는 `IRefCountedBufferSource`를 받을 수 있다.
- `IoUringRegisteredPayloadBlockPool`은 fixed table slot 수만큼만 대여하고 full 상태에서는 hidden allocation 을 만들지 않는다.
- `IoUringCompositePayloadBufferSource`는 registered pool hit 를 우선하고 miss 때 fallback source 로 간다.
- `BrokerServer`는 concrete backend 를 보지 않고 `ITransportPayloadBufferSourceProvider` optional seam 으로 TCP payload source 를 받는다.
- `IoUringTransport`는 registered payload pool 목록에서 backing array fixed index 를 찾아 `SendFixedRegisteredPayloadAsync(...)`를 먼저 시도하고,
  miss 때 기존 baseline `SendArrayAsync(...)`로 fallback 한다.

## D229 - registered payload pool 구현은 BrokerServer concrete backend 의존 없이 transport payload source provider seam 으로 연결한다.

- Date: 2026-07-09
- Status: Proposed
- Scope: Phase 6 `io_uring` registered payload pool implementation plan.

### Context

D226 설계는 `TcpFrameAssembler`가 registered payload source 에서 block 을 대여할 수 있게 하는 방향을 제안했다.
하지만 실제 코드에서 `TcpFrameReceiveHandler`는 `BrokerServer`가 직접 생성하고, `BrokerServer`는 현재
`PinnedBlockMemoryPool`만 알고 있다. `IoUringTransport`가 registered payload pool 을 갖더라도
Server 가 concrete `IoUringTransport`를 직접 캐스팅하면 OS별 backend 가 `ITransport` 뒤에 숨는다는 아키텍처 규칙이 깨진다.

### Decision

D229 구현 계획은 backend-neutral `ITransportPayloadBufferSourceProvider` seam 을 추가하는 방향으로 잡는다.

- `ITransportPayloadBufferSourceProvider`는 `Hps.Transport` abstraction 으로 둔다.
- `BrokerServer.StartTcpAsync`는 transport start 이후 receive handler 를 만들고,
  transport 가 provider 를 구현하면 provider 에 fallback `PinnedBlockMemoryPool`을 넘겨 source 를 받는다.
- SAEA/RIO/default 경로는 fallback pool source 를 그대로 사용한다.
- `IoUringTransport`는 provider 구현으로 queue lifetime 에 묶인 registered payload pool 과 fallback source 를 합성한다.
- registered payload pool 은 connection resource 가 아니라 transport/source lifetime 에 묶는다.
- send helper 는 transport-owned registered payload pool 목록에서 backing array fixed index 를 조회한다.

구현 계획:
`docs/superpowers/plans/2026-07-09-iouring-registered-payload-pool.md`

### Rationale

이 seam 은 `BrokerServer`가 backend concrete type 을 알 필요 없이 TCP assembler source 를 backend 가 제안할 수 있게 한다.
동시에 SAEA/RIO와 기존 public host 사용자는 계속 같은 constructor 와 fallback pool 을 사용한다.

registered payload pool 을 connection resource 에 묶으면 TCP receive source 와 send helper 간 소유 경계가 어긋난다.
source 가 만든 block 이 어떤 connection resource 로 send 될지는 fan-out 시점에 결정되므로,
transport/source lifetime 에 pool 을 두고 send helper 가 backing array identity 로 lookup 하는 편이 단순하다.

### Consequences

- 구현은 Buffers -> Protocol -> IoUring pure pool -> Server provider seam -> fixed send opt-in 순서로 진행한다.
- 새 타입 Red 는 compile failure 가 아니라 reflection shape assertion failure 로 시작한다.
- Linux native fixed payload hit 여부는 local shape tests 가 아니라 `iouring-linux-contract.yml` artifact 로 확인한다.
- UDP registered send pool, default backend promotion, zero-copy 달성 주장, latency hard gate 승격은 계속 제외한다.

## D226 - production TCP fixed-write 연결 전, TCP-first registered payload pool 설계로 owner/source 경계를 먼저 둔다.

- Date: 2026-07-09
- Status: Proposed
- Scope: Phase 6 `io_uring` production fixed-write follow-up.

### Context

D224/D225 이후에도 production TCP payload 는 아직 fixed table 에 안정적으로 연결되지 않았다.
현재 publish payload 는 `TcpFrameAssembler`가 `PinnedBlockMemoryPool.RentCounted()`로 대여한
`RefCountedBuffer`에 TCP frame payload 를 1회 복사하고, broker fan-out 이 같은 buffer 를 구독자별 ref 로 공유한다.
이 block 은 publish 시점에 동적으로 선택되므로 connection resource 생성 시점의 fixed send registry 에 미리 넣을 수 없다.

`IoUringFixedSendBufferRegistry`는 backing `byte[]` identity 를 fixed index 로 조회하는 shape 를 갖지만,
그 lookup 이 hit 하려면 payload block 자체가 queue/resource lifetime 에 묶인 registered block source 에서 나와야 한다.

### Decision

production TCP payload `WRITE_FIXED` default 연결 전에 TCP-first registered payload pool 을 설계한다.

설계 문서:
`docs/superpowers/specs/2026-07-09-iouring-registered-payload-pool-design.md`

핵심 방향:

- `RefCountedBuffer`의 concrete `PinnedBlockMemoryPool` 반환 경계를 `IRefCountedBufferOwner`로 일반화한다.
- `TcpFrameAssembler`의 concrete pool 대여 경계를 `IRefCountedBufferSource`로 일반화한다.
- `Hps.Transport.IoUring` 내부에 queue/resource lifetime 에 묶인 `IoUringRegisteredPayloadBlockPool`을 둔다.
- registered pool 은 hidden allocation fallback 을 하지 않고, fallback 은 composite source 가 기존 `PinnedBlockMemoryPool`을 명시적으로 선택한다.
- TCP assembler 는 registered source 에 payload 1회 복사를 흡수할 수 있다.
- UDP receive path 는 endpoint receive slot/window ownership 과 결합되어 있으므로 첫 범위에서 제외한다.
- payload registry hit 는 fixed helper path 를 탈 수 있고, miss 는 baseline `SendArrayAsync(...)`로 fallback 한다.

### Rationale

send hot path 에서 per-send `RegisterBuffers`/`UnregisterBuffers`를 수행하면 D210 계열 timeout/hang 위험으로 되돌아간다.
반대로 queue lifetime 에 묶인 registered payload block source 를 두면 payload block identity 와 fixed index 관계가
send 전에 안정적으로 정해진다.

TCP는 이미 D009/D010에 따라 receive chunk 에서 owned payload block 으로 1회 복사하므로,
그 복사를 registered block 으로 흡수하기 쉽다.
UDP는 receive slot 을 kernel receive target 으로 직접 사용하므로 같은 설계에 섞으면 use-after-free,
receive window depth, fan-out ref lifetime, fixed send table slot 정책을 한 번에 풀어야 한다.

### Consequences

- 다음 구현 전 단계는 D226 설계 검토다.
- 설계 승인 뒤 구현 계획은 owner abstraction, source abstraction, registered payload pool contract,
  TCP assembler integration, io_uring send opt-in integration 으로 나눠야 한다.
- production TCP payload send path 는 검토 전까지 계속 baseline `SendArrayAsync(...)`다.
- UDP registered send pool 연결, default backend promotion, zero-copy 달성 주장,
  latency hard gate 승격은 계속 제외한다.

## D205 - io_uring transport 는 close 로 unregister 된 connection 의 TCP send pump task 도 shutdown 때 기다린다.

- Date: 2026-07-07
- Status: Accepted
- Scope: Phase 6 `io_uring` TCP send pump task lifetime.

### Context

D204 수정 후 사용자 push 기준 `iouring-linux-contract.yml` run `28841586637`을 실행했지만,
동일하게 `TcpLoopback_WhenIoUringAvailable_SendsQueuedPayloadToPeer`가 pool `RentedCount` expected 0 / actual 1로 실패했다.
TRX 기준 fixed-send lease native evidence 는 계속 capability `Available` 상태로 통과했다.

추가 분석 결과 테스트는 payload 수신 뒤 `server.Close()`를 먼저 호출한다.
이 시점에 `TransportConnection`은 `_connections` 목록에서 unregister 되므로,
`StopAsync`가 open connection snapshot 만 보고 in-flight drain을 기다리는 방식으로는 이미 닫혔지만 아직 unwind 중인 send pump 를 볼 수 없다.

### Decision

`IoUringTransport`는 자신이 시작한 TCP send pump `Task`를 connection list 와 별도로 추적한다.
`StartSendLoop`는 task 를 등록하고, task completion continuation 은 목록에서 제거한다.
`StopAsync`/`Dispose`는 `StopCore()`에서 얻은 send pump task snapshot 을 기다린 뒤 TCP in-flight send drain 과 native owner dispose 를 수행한다.

### Rationale

connection registry 는 endpoint diagnostics 와 open connection 수명 관리를 위한 목록이다.
반면 send pump task 는 close 이후에도 finally/unwind 를 통해 transport-owned send ref 를 반환해야 하는 background work 다.
두 수명은 다르므로 shutdown 완료 관측에는 connection snapshot 이 아니라 pump task snapshot 도 필요하다.

### Consequences

- `server.Close()`로 이미 unregister 된 connection 의 send pump finally 도 `StopAsync` 완료 전에 관측된다.
- completion loop 와 queue dispose 는 send pump task 대기 이후로 지연된다.
  그래야 close로 발생한 CQE가 send context waiter 를 깨우고 pump finally 가 ref 를 반환할 수 있다.
- 이 결정은 TCP send pump shutdown ownership 보강이며, production fixed-write pump 연결이나 zero-copy send 를 열지 않는다.

## D204 - io_uring StopAsync 는 connection close 이후 TCP in-flight send ref 반환 완료를 기다린다.

- Date: 2026-07-07
- Status: Accepted
- Scope: Phase 6 `io_uring` TCP send shutdown ownership.

### Context

사용자 push 이후 `iouring-linux-contract.yml` run `28840613527`을 실행했다.
새 `Lease_WhenLinuxCapabilityAvailable_WritesRegisteredPayloadSliceToSocketPair`는 capability `Available` 상태에서 통과했지만,
기존 `TcpLoopback_WhenIoUringAvailable_SendsQueuedPayloadToPeer`가 pool `RentedCount` expected 0 / actual 1로 실패했다.

payload 는 peer socket 에 도착했지만, `IoUringTransport.StopAsync`가 connection close 직후 반환되면서
send pump 의 `InFlightSend.Dispose()` finally 경로가 transport-owned ref 를 반환하기 전에 pool leak 단언이 실행될 수 있었다.

### Decision

`TransportConnection`은 pending queue 에서 dequeue 된 in-flight send count 를 추적하고,
`WaitForInFlightSendsToDrainAsync()`로 모든 in-flight send ref 반환 완료를 관측할 수 있게 한다.

`IoUringTransport.StopAsync`는 `StopCore()`로 listener/connection/endpoint close 를 시작한 뒤,
snapshot 된 TCP connection 들의 in-flight send drain task 를 기다린다.

### Rationale

`Close()`가 in-flight ref 를 즉시 반환하면 커널 send 가 아직 payload 를 참조할 수 있는 상황에서
buffer 가 pool 로 돌아갈 수 있다. 따라서 close 는 pending queue 만 drain 하고,
이미 dequeue 된 ref 는 pump completion/finally 가 반환한다는 기존 ownership 규칙을 유지해야 한다.

다만 async `StopAsync`는 shutdown 완료 관측 지점이므로, pool leak 검증과 다음 lifecycle 단계가
pump finally와 경쟁하지 않도록 drain 완료를 기다리는 것이 맞다.

### Consequences

- `IConnection.Close()`의 동기 의미는 유지된다. close 는 pending drain 과 resource close 를 수행하지만,
  in-flight send ref 는 pump completion/finally 경로에서 반환된다.
- `IoUringTransport.StopAsync`는 TCP send in-flight ref 반환 완료를 기다리므로,
  Linux remote gate 의 shutdown 직후 pool count race 를 줄인다.
- production TCP pump fixed-write 연결, zero-copy send, default promotion 은 여전히 별도 remote gate 이후 판단한다.

- shutdown 전 registry 에 없는 token 은 기존처럼 `InvalidOperationException`으로 실패시켜 실제 token routing bug 를 숨기지 않는다.

### Rationale

Linux kernel 은 socket close/cancel/stop 경합에서 이미 완료됐거나 취소된 CQE 를 resource unregister 이후에 노출할 수 있다.
이 CQE는 새 managed context 로 라우팅하면 안 되지만, transport shutdown 을 실패시킬 오류도 아니다.
반대로 정상 운영 중 unknown token 을 항상 무시하면 user_data/token routing 손상을 감출 수 있으므로,
무시 정책은 shutdown 이후로만 제한해야 한다.

### Consequences

- TCP baseline open-loop stop 중 stale CQE가 남아도 process abort 로 이어지지 않아야 한다.
- 정상 운영 중 token corruption 은 계속 즉시 실패한다.
- 다음 작업은 fix push 이후 원격 `iouring-benchmark-artifacts.yml` artifact gate 를 재실행해
  TCP baseline exit 134 재발 여부와 TCP/UDP envelope signal-count 0을 확인하는 것이다.

## D147 — Linux io_uring benchmark artifact 는 별도 workflow_dispatch workflow 로 수집한다.

- Date: 2026-07-01
- Status: Accepted
- Scope: Phase 6 `io_uring` benchmark evidence path.

### Context

D146으로 benchmark CLI 는 `--backend iouring`을 지원하게 됐다.
하지만 기존 `.github/workflows/benchmark-artifacts.yml`은 Windows SAEA benchmark artifact 용이고,
`.github/workflows/iouring-linux-contract.yml`은 Linux `Hps.Transport.IoUring.Tests` 계약 검증용이다.
따라서 Linux available runner 에서 TCP/UDP `io_uring` benchmark raw report 를 생성하는 원격 evidence path 가 없다.

### Decision

`.github/workflows/iouring-benchmark-artifacts.yml`을 별도 workflow 로 추가한다.

- trigger 는 `workflow_dispatch` 전용으로 둔다.
- runner 는 `ubuntu-latest`로 둔다.
- runner identity 는 `ci-linux-iouring-x64-01`, kind 는 `ci`로 둔다.
- TCP와 UDP를 같은 workflow 에서 모두 실행하되 output root 는 protocol 별로 분리한다.
- protocol 별 root 바로 아래에 날짜 directory 를 둔다.
  즉 `runner/tcp/<yyyy-mm-dd>/session-01`, `runner/udp/<yyyy-mm-dd>/session-01` 구조를 사용한다.
  `BaselineHistoryReader`가 parent root 바로 아래 날짜 directory 만 읽기 때문이다.
- 각 protocol 은 `--baseline-suite <session-dir> --runs 1 --protocol <tcp|udp> --backend iouring`을 실행한다.
- 각 protocol 별 `summary.json`/`summary.md`와 `history.json`/`history.md`를 생성한다.
- root `summary.md`와 `dotnet-info.txt`를 함께 업로드한다.
- baseline/summary/history 단계의 non-zero exit code 는 저장해 두고, upload 이후 final gate 에서 job failure 로 복원한다.

### Rationale

기존 Windows benchmark workflow 를 matrix 로 넓히면 SAEA benchmark, CI baseline artifact, Linux `io_uring` evidence 가
같은 trigger와 artifact 정책에 섞인다. 현재 필요한 것은 default promotion 이 아니라 D146 selector 가 Linux runner 에서
실제 benchmark raw report 를 만들 수 있는지 확인하는 것이다. 별도 수동 workflow 가 가장 작고 추적 가능한 경계다.

### Consequences

- `io_uring` benchmark evidence 는 push 이후 사용자가 수동 workflow 를 실행해야 생긴다.
- artifact 는 `iouring-benchmark-artifacts-<yyyy-mm-dd>-github-<run-id>-<attempt>` 이름으로 올라간다.
- artifact 내부 history root 는 `tcp/`와 `udp/`이며, 각 root 아래 날짜 directory 와 session directory 가 위치한다.
- latency/HWM/warning 값은 report data 이며 hard gate 나 warning-as-failure 로 승격하지 않는다.
- default backend selection 은 계속 변경하지 않는다.
- fixed registration, zero-copy send, IPv6 direct UDP 는 이 artifact 를 검토한 뒤 별도 설계로 판단한다.
