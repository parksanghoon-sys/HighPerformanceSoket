# 검토: Phase 3 — 브로커 pub/sub 라우팅 동시성 (설계 검증)

- **대상**: PLAN Phase 3 의 `topic → 구독자 set` 라우팅 테이블 **설계** (코드 전).
  설계 후보들을 프로토타입으로 구현해 동시성 스트레스로 실측했다(`scratch/VerifyRouting`, 검토 후 삭제).
- **요약 판정**: **must — 빈 토픽 정리 경합 주의.** 아래 R1 을 지키는 설계로 확정.

## 핵심 발견 (R1) — "빈 토픽 정리 vs 동시 구독" 경합으로 구독이 유실된다
구독자 set 이 비면 토픽 엔트리를 제거하는 흔한 최적화는 **동시 구독을 유실**시킨다. 실측:

| 라우터 설계 | 유실 경합(20만 회) | churn |
|---|---|---|
| **Naive** eager-cleanup (`set.IsEmpty 면 dict.TryRemove(topic)`) | ❌ **약 51% 유실** | — |
| lock-free **verify-retry** (영리하게 고치려 시도) | ❌ **약 50% 유실 (여전히 틀림)** | — |
| **NoCleanup** (빈 set 을 제거하지 않음) | ✅ **0 유실** | ✅ 예외 0, 누수 0 |
| **LockedCleanup** (set 인스턴스 락으로 추가/빈-제거 직렬화) | ✅ **0 유실** | ✅ 예외 0, 누수 0 |

> 경합 시나리오: 토픽 T 에 X 가 있는 상태에서 "Y 구독"과 "X 해지(→빈 정리 유발)"가 동시에 일어나면,
> Y 가 *방금 제거된* set 에 추가되어 라우팅 테이블에서 사라진다.
>
> ⚠️ **교훈 2가지**:
> 1. 흔한 churn 테스트(같은 스레드가 구독→해지 반복)는 이 버그를 **놓친다**(예외 0·누수 0 으로 통과).
>    반드시 **"동시 구독 vs 빈-정리" 타깃 경합 테스트**가 필요하다.
> 2. **영리한 lock-free 정리는 실측에서 틀렸다.** 단순·정확한 쪽을 택할 것.

## 권장 설계
둘 중 하나(둘 다 실측 0 유실):
- **(권장) NoCleanup + 주기적 안전 sweep**: 빈 토픽 엔트리를 즉시 제거하지 않는다. 토픽 키 누적이
  문제될 때만 별도 sweep 이 **"비어 있고 일정 시간 미사용"인 엔트리를 락/검증 후** 제거한다.
  구현이 가장 단순하고 핫패스(구독/해지)에 락이 없다.
- **LockedCleanup**: 즉시 정리가 필요하면, **구독자 set 인스턴스를 `lock`** 으로 잡아 "추가"와
  "빈-제거"를 직렬화한다. Subscribe 는 락 안에서 set 이 아직 매핑돼 있는지 확인 후 추가하고, 아니면 재시도.

## 발행(Publish) 경로 메모
- Publish 는 `topic` set 을 순회하며 각 구독자 송신 큐에 `(RefCountedBuffer, off, len)` enqueue(복사 0회).
  `ConcurrentDictionary` 순회는 mutation 과 동시에도 예외 없이 동작함을 확인(churn 테스트).
- 다중 발행 스레드가 같은 구독자에 동시 enqueue 하는 문제는 [phase2-transport-bipbuffer.md] D1
  ("MPSC 큐 → 단일 펌프 → SPSC 송신 BipBuffer")에서 다룬다.

## 반드시 추가할 테스트
- **R1 회귀(타깃 경합)**: T 에 X 존재 상태에서 "Y 구독"‖"X 해지" 동시 실행을 수만~수십만 회 반복,
  매회 Y 가 T 에 남아 있는지 단언. 정리 로직이 틀리면 즉시 대량 실패해야 한다.
- 고동시성 churn + 동시 publish: 예외 0, 최종 구독자 0(누수/불균형 없음).
- (정리 도입 시) 빈 sweep 이 활성 구독을 지우지 않는지.
