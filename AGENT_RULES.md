# AGENT_RULES.md — 실행 상태 관리 규칙

이 문서는 `AGENTS.md`의 핵심 규칙을 대체하지 않는다. `AGENTS.md`는 불변식과 코딩 규칙의 최상위 기준이고,
이 문서는 매 작업 사이클에서 어떤 상태 파일을 읽고 갱신할지 정리하는 실행용 보조 규칙이다.

## 현재 목표
- 고성능 TCP/UDP pub/sub 메시지 브로커를 구현한다.
- 우선 목표 부하는 **4096 bytes 메시지 100 Hz**를 지연 누적 없이 처리하는 것이다.
- 이 목표는 최소 기준선이다. Phase 4 이후에는 p50/p99 지연, 처리량, 큐 적체 여부로 검증 가능한 수치로 구체화한다.

## 상태 파일 역할
- `PLAN.md`: 전체 Phase 순서와 큰 구현 범위.
- `CURRENT_PLAN.md`: 지금 바로 이어서 실행할 단일 작업 단위와 검증 방법.
- `TODOS.md`: 현재 실행 가능한 TODO, 지연된 backlog, 완료 이력.
- `CHANGELOG_AGENT.md`: 각 작업 사이클에서 실제로 변경한 내용과 검증 결과.
- `DECISIONS.md`: 이후 구현에 영향을 주는 결정과 근거.
- `.claude/review/`: Claude 검토 결과. must-fix는 다음 구현 작업 전에 해소한다.

## 작업 단위와 사용자 리뷰 게이트
- 구현은 항상 **기능별로 작고 리뷰 가능한 단위**로 나눈다. 서로 독립적인 기능, 보강 테스트, 리팩터링,
  문서 정리는 한 번에 묶지 말고 별도 사이클로 분리한다.
- 한 사이클에서는 `CURRENT_PLAN.md`에 적힌 단일 작업 단위만 완료한다. 진행 중 범위가 커지면 즉시 멈추고
  현재 단위를 더 작게 쪼개며, 뒤로 미룬 항목은 `TODOS.md`의 `Deferred Backlog`에 self-contained하게 남긴다.
- 기능 단위가 구현·검증·상태 문서 갱신까지 끝나면 다음 기능으로 자동 진행하지 않는다. 사용자가 검토한 뒤
  계속 진행하라고 명시하기 전까지 다음 구현 사이클을 시작하지 않는다.
- 각 기능 단위는 검증과 상태 문서 갱신을 마친 뒤 **해당 단위의 관련 파일만 stage 하여 단일 커밋으로 남긴다**.
  서로 다른 기능, 보강 테스트, 리팩터링, 문서 정리를 같은 커밋에 섞지 않는다.
- 커밋 전에는 `git status`로 의도하지 않은 변경이 포함되지 않았는지 확인한다. 이미 존재하던 unrelated 변경이나
  사용자가 만든 변경은 stage 하지 않고, 현재 단위와 분리할 수 없으면 커밋 전에 사용자에게 확인한다.
- 예외는 빌드 실패, 방금 만든 테스트 실패, 문서 상태 불일치처럼 현재 기능 단위를 완료하기 위해 반드시
  바로 고쳐야 하는 문제로 제한한다. 이 경우에도 수정 이유와 검증 결과를 `CHANGELOG_AGENT.md`에 남긴다.

## 테스트 작성 규칙
- 테스트에는 **무엇을 검증하는지와 왜 필요한지**를 한국어 주석으로 남긴다. 단순히 테스트 이름을 반복하지 말고,
  보호하려는 불변식, 회귀 사례, 경계 조건, 동시성/소유권 가정처럼 다음 작업자가 테스트 의도를 이해하는 데
  필요한 정보를 짧게 적는다.
- 테스트 주석은 각 테스트 메서드 바로 위에 둔다. helper 메서드는 여러 테스트의 실패 진단이나 참조 모델 역할처럼
  의도가 비자명할 때만 주석을 추가한다.
- Red 테스트를 추가할 때도 주석을 먼저 함께 작성해, 실패가 어떤 요구사항을 증명하는지 드러나게 한다.

## 사이클 시작 순서
1. `AGENTS.md`를 읽어 불변식과 코딩 규칙을 확인한다.
2. `PLAN.md`를 읽어 Phase 순서를 확인한다.
3. `CURRENT_PLAN.md`와 `TODOS.md`를 읽어 다음 실행 지점을 확인한다.
4. `CHANGELOG_AGENT.md`와 `DECISIONS.md`의 최신 항목을 확인한다.
5. `.claude/review/`에 unresolved must-fix가 있는지 확인한다.
6. 실제 파일 상태와 문서 상태가 충돌하면 문서보다 실제 파일을 우선 확인하고, 차이를 상태 문서에 기록한다.

## 사이클 종료 순서
1. `CURRENT_PLAN.md`를 다음 실행 지점으로 갱신한다.
2. 완료한 항목은 `TODOS.md`의 `Completed`로 이동한다.
3. 즉시 처리하지 못한 후속 작업은 `TODOS.md`의 `Deferred Backlog`에 self-contained하게 기록한다.
4. `CHANGELOG_AGENT.md`에 변경 사항, 검증 명령, 남은 불확실성을 기록한다.
5. 구조, 계약, 목표, Phase 순서에 영향을 주는 선택은 `DECISIONS.md`에 남긴다.

## 검증 원칙
- `dotnet test`가 성공해도 테스트가 0개 discover된 경우 완료로 보지 않는다.
- Phase 완료는 테스트 실행 성공, 필수 테스트 케이스 존재, 누수 0 확인, Claude must-fix 해소를 함께 만족해야 한다.
- 성능 목표는 Phase 4 전에는 설계 제약으로만 취급하고, Phase 4 이후에는 재현 가능한 벤치마크 수치로 관리한다.
